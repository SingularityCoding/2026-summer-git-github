# 为什么需要 Git

在真正敲下第一条 Git 命令之前，先花几分钟想清楚一件事：Git 到底是用来干什么的，为什么我们不能就这么随便应付过去。

## 一个具体场景

你正在写一个小项目，今天改了不少代码。后来程序突然跑不起来了，你已经记不清具体改了哪些地方；你想退回到昨天还能跑的版本，又想留住今天写的一部分有用的东西。再往前想一步：如果是两个人一起改同一份代码，怎么知道谁改了什么、该怎么合并、出了问题该找谁负责？

这些问题，靠"手动复制文件夹"是解决不了的——`project_final.zip`、`project_final_v2.zip`、`project_final_v2_真的最终版.zip` 这种命名迟早会把自己绕晕。

从这个场景可以引出一句很关键的话：我们需要一个专门管理代码变化的工具。这个工具真正要解决的核心问题，不是"保存文件"，而是**记录项目随时间变化的历史**——让我们随时能看差异、保存阶段性成果、回到过去的版本，并且支持多人协作。

## 版本控制走过的三个阶段

这里简单展开一下版本控制系统（version control system）的发展史，帮大家建立一点点背景。不需要深究细节，有个大致印象就够，后面动手敲命令的过程会让这个印象慢慢变得具体。

1. **手动复制文件**：最朴素的做法，每次改动前复制一份文件夹，`project_final`、`project_final_v2`……问题很明显：混乱、不可追踪、几乎没法协作。谁改了什么、什么时候改的、为什么这么改，全靠脑子记，忘了就没了。
2. **集中式版本控制系统（centralized version control system）**：例如 CVS、Subversion（SVN）。这类系统通常依赖一个中心服务器，团队成员从服务器上 checkout、update、commit。比"复制文件夹"靠谱太多了，但对中心服务器依赖较重——离线能力和分支体验都比较有限，连不上服务器基本就干不了活。
3. **分布式版本控制系统（distributed version control system）**：Git 走的就是这条路。每个开发者本地都有一份完整的仓库历史，可以离线提交、离线查看日志、随时创建分支；远程仓库更多是用来同步和协作，而不是唯一的历史来源。

讲到这里就够了，不需要展开 Git 内部到底是怎么实现的。这一部分现在没完全理解也没关系，这只是给一个大致背景，后面的命令和操作会让你慢慢建立起真正的直觉。
